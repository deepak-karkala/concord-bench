# Concord v0.1 Engineering Plan

Generated by /plan-eng-review on 2026-04-25

## 1. Locked Architectural Decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| 1.1 | AWM integration | Partial reuse | Use AWM for scenario/task/DB/sample. Concord-native env+verifier. |
| 1.2 | Env runtime | Hybrid | Concord-native turn-based core + thin PettingZoo AEC wrapper. |
| 1.3 | Activation logging | nnsight | Lingua franca of AISI/Goodfire/EleutherAI interp work. |
| 1.4 | Cultural variants | LLM adaptation + audit | Adapt scenarios using Hofstede/Hall frameworks, then native-speaker audit. |
| 1.5 | Test framework | pytest + pytest-asyncio | Standard Python research stack. |
| 1.6 | Storage | SQLite per-scenario, JSONL transcripts, Parquet aggregates | Per AWM pattern. |
| 1.7 | Offer extraction | Constrained generation + regex fallback | Test all frontier models in week 1. |
| 1.8 | Repeated-game state | SQLite table keyed on (scenario_id, buyer_model, seller_model) | Schema-versioned, validated on load. |
| 1.9 | Reproducibility | Pin model snapshot dates + log prompt hashes + version-tag results | Required before any public results. |
| 1.10 | Cost management | Cache LLM calls keyed on (model, prompt_hash, temp, seed) | ~60% cost reduction on re-runs. |

## 2. AWM Reuse Strategy

```
AWM PIPELINE                       CONCORD REUSE
────────────────                   ─────────────────────────────────────
gen scenario     ────────▶          ✓ AS-IS (platform-level seeds)
gen task         ────▶ adapt ──▶    Modify prompt for negotiation role-pairs
gen db           ────────▶          ✓ AS-IS (scenario state schema)
gen sample       ────────▶          ✓ AS-IS (initial state population)
gen spec         ────▶ adapt ──▶    Modify for offer-schema spec instead of API
gen env          ────────▶          ✗ NOT REUSED (Concord env is 2-agent, not REST)
gen verifier     ────▶ adapt ──▶    One signal in multi-objective grader
```

Reuse ratio: ~70% of AWM pipeline.

### Integration pattern: PyPI optional dependency, not vendored

AWM is published to PyPI as `agent-world-model` (v0.1.0 as of 2026-04). Concord lists it as an **optional dependency** under the `[synth]` extra, NOT vendored inside the repo.

**Why optional:** the synthesis pipeline is a build-time dependency for the Concord author and a small minority of users extending the benchmark. Most end users (~95%) are researchers who want to evaluate models against pre-generated Concord scenarios. They don't need AWM, scenario generation, or its transitive dependencies (LLM API clients for synthesis, FastAPI, SQLAlchemy ORM, etc.).

**Why PyPI dep, not vendored subtree:**
- Vendoring forces every downstream consumer to pull AWM and its deps even if they never call synthesis code
- AWM is actively developed at Snowflake; tracking upstream via PyPI version pinning is cleaner than git subtree pulls
- Standard practice for Python ML libraries (Transformers, LangChain, SB3) is optional extras for heavy deps
- PyPI dependency works for users who can't easily clone GitHub (corporate networks, offline mirrors)

**Concord's pyproject.toml will declare:**

```toml
[project]

    "pydantic>=2.0",
    "pettingzoo>=1.24",
    "huggingface-hub>=0.20",     # for scenario download
    "anthropic>=0.30",

    "google-genai>=0.5",

[project.optional-dependencies]

    "agent-world-model>=0.1.0",  # pulled only when generating scenarios
```

**Install paths for end users:**

```bash
pip install concord-bench              # 95% of users (just evaluating models)
pip install concord-bench[synth]       # extending the benchmark
pip install concord-bench[interp]      # white-box research
pip install concord-bench[all]         # everything
```

**Concord code calls AWM exactly the same way regardless of install path:**

```python
# concord/concord/synth/enrichment.py
from awm.core.scenario import generate_scenarios
from awm.core.task import generate_tasks_for_scenario

def enrich_for_negotiation(awm_scenario, domain, culture):
    """Transform AWM scenario output → Concord scenario YAML."""
    ...
```

The import will fail clearly with `ModuleNotFoundError: No module named 'awm'` if a user tries to call synthesis code without the `[synth]` extra. Users get a clear error path: install the extra. Synthesis modules should be the only files importing AWM, and they should import lazily where possible so a user without `[synth]` installed can still load `concord/__init__.py`.

### Author dev environment

For Concord v0.1 development, install AWM normally from PyPI:

```bash
cd concord/
uv sync                               # pulls in main deps
uv sync --extra synth                  # adds agent-world-model from PyPI
uv sync --extra synth --extra interp   # everything you need to build v0.1
```

If you need to inspect AWM source or modify it during dev, drop to editable install of your local clone:

```bash
uv pip install -e ../agent-world-model
```

Use editable only when you need to read/modify AWM internals. Default to PyPI install.

## 3.5 Scenario Distribution

The scenarios themselves are NOT bundled in the pip package beyond a small seed set. Concord scenarios are distributed through Hugging Face Hub as a versioned dataset. This is the same pattern used by BIG-bench, MMLU, HELM, and every other major LLM benchmark.

### Why HF Hub instead of bundling

| Concern | Bundle in pip | HF Hub dataset |
|---|---|---|
| Package size | 5K scenarios × ~5KB = ~25MB pip wheel | <1MB pip wheel |
| Network on first use | None | Required once, cached after |
| Scenario updates | Requires Concord version bump + pip release | Independent dataset version, no code release |
| Versioning | Tied to Concord code version | Independent (`concord-bench` v0.1.0 can pull `concord-scenarios` v0.1.0, v0.1.1, ...) |
| Discoverability | Hidden inside pip package | Searchable on HF, viewable in browser, has dataset card |
| Standard pattern | Few benchmarks do this | Every benchmark does this |

### What ships where

```
PIP PACKAGE (concord-bench on PyPI)
  - concord/data/seed_yamls/{domain}/    ~50 hand-authored seed scenarios
                                          (small, included so users can smoke-test offline)

HUGGING FACE DATASET (concord-bench on HF)
  - data/scenarios/v0.1.0/               ~5K generated scenarios (4 domain × 5 culture × variants)
  - data/scenarios/v0.1.0/repeated/      Repeated-game sequences
  - data/calibration/                    50 hand-labeled transcripts (for grader calibration tests)
  - data/reference_results/              v0.1 frontier model results (for replication checks)
  - README.md                            Dataset card with provenance, schema, license
```

### Loader pattern

```python
# concord/concord/data/loader.py
from huggingface_hub import snapshot_download
from pathlib import Path
import yaml
from concord.schemas.scenario import Scenario

DATASET_REPO = "deepakkarkala/concord-bench"  # placeholder; real handle TBD

def load_scenarios(
    version: str = "v0.1.0",
    domain: str | None = None,
    culture: str | None = None,
    cache_dir: Path | None = None,
) -> list[Scenario]:
    """Pull scenarios from HF Hub, cache locally, parse YAMLs into Pydantic objects."""
    local_path = snapshot_download(
        repo_id=DATASET_REPO,
        repo_type="dataset",
        revision=version,
        cache_dir=cache_dir,
        allow_patterns=[f"data/scenarios/{version}/**/*.yaml"],
    )
    scenario_dir = Path(local_path) / "data" / "scenarios" / version
    scenarios = []
    for yaml_path in scenario_dir.rglob("*.yaml"):
        if domain and yaml_path.parent.parent.name != domain:
            continue
        if culture and yaml_path.parent.name != culture:
            continue
        with yaml_path.open() as f:
            scenarios.append(Scenario.model_validate(yaml.safe_load(f)))
    return scenarios


def load_seeds() -> list[Scenario]:
    """Bundled seed scenarios. Works offline. ~50 scenarios."""
    seed_dir = Path(__file__).parent / "seed_yamls"
    return [
        Scenario.model_validate(yaml.safe_load(p.read_text()))
        for p in seed_dir.rglob("*.yaml")
    ]
```

End-user code:

```python
# Researcher evaluating their model

from concord.runners import run_episode

scenarios = load_scenarios(version="v0.1.0", domain="saas_procurement", culture="JP")
for scenario in scenarios:
    result = run_episode(scenario, model="claude-opus-4-7")
    ...
```

### Versioning protocol

- Concord package versions and HF dataset versions can drift independently.
- Each Concord release pins a *minimum compatible* dataset version in `concord/data/loader.py`.
- Dataset versions are git tags on the HF dataset repo (semver: v0.1.0, v0.1.1).
- Re-running a paper's results requires both `concord-bench==X.Y.Z` and `concord-bench` HF dataset at the corresponding version. Document both in published results.

### What the Concord author does at v0.1 launch

1. Generate 5K+ scenarios locally via `concord generate` (this calls AWM via the `[synth]` extra).
2. Audit a sample manually.
3. Push the dataset to HF: `huggingface-cli upload deepakkarkala/concord-bench data/scenarios/v0.1.0`.
4. Tag the dataset version: `huggingface-cli tag deepakkarkala/concord-bench v0.1.0`.
5. Bump `concord-bench` package version to 0.1.0 with the dataset version pinned.
6. Publish to PyPI.

Now anyone running `pip install concord-bench` and `concord run` gets the v0.1 scenarios automatically pulled from HF on first use.

### Optional bundled split

If users complain about HF dependency (corporate networks blocking it, offline labs, etc.), add an optional `[scenarios]` extra that bundles all 5K scenarios. Cost: 25MB extra wheel size. Don't do this in v0.1; add only if asked.

## 6. Failure Modes Registry

| # | Codepath | Failure mode | Test? | Error handling? | User sees |
|---|---|---|---|---|---|
| F1 | AWM upstream prompt change | Concord enrichment breaks silently | ✓ schema test | ✓ pydantic raises | clear error |
| F2 | Frontier API rate limit | Episodes fail mid-batch | ✓ unit | ✓ retry+backoff | logged warning, episode retried |
| F3 | Open-weight model OOM | Activation logging crashes | ✗ untested | ✗ unhandled | **CRITICAL GAP** — silent crash |
| F4 | Constrained generation unsupported | Offer parsing falls to regex | ✓ unit | ✓ regex fallback | logged, episode continues |
| F5 | Calibration set drift over time | LLM judge accuracy decays unnoticed | ✗ untested | ✗ no monitoring | **CRITICAL GAP** — silent degradation |
| F6 | Cultural audit reviewer disagreement | Some scenarios marked inauthentic | ✓ audit log | ✗ no resolution flow | scenario flagged, manual review needed |
| F7 | Repeated-game state corruption | Round N can't load round N-1 state | ✗ untested | ✗ unhandled | **CRITICAL GAP** — silent reset |

### Critical Gap Mitigations (must implement before launch)

**F3 — Open-weight OOM:**
```python
# concord/agents/open_weight_adapter.py
try:
    activations = extract(model, batch_size=current_size)
except torch.cuda.OutOfMemoryError:
    log.warning("OOM at batch_size=%d, halving", current_size)
    torch.cuda.empty_cache()
    if current_size <= 1:
        raise OpenWeightOOMError("Cannot extract at batch_size=1")
    return extract_with_retry(model, batch_size=current_size // 2)
```

**F5 — Calibration drift:**
- Schedule weekly cron: re-run 10 random samples through LLM judges
- Compute Cohen's kappa against author labels
- Alert (Slack webhook / email) if kappa drops below 0.7
- Implementation: `concord/graders/calibration.py::monitor_drift()`

**F7 — Repeated-game state corruption:**
- Add `schema_version` integer to every persisted state row
- Validate on load: raise `StateSchemaMismatchError` if mismatch
- Migration script for any schema bump
- Implementation: `concord/env/state.py::load_state()`

## 7. Performance Budget

### 7.1 LLM API costs (the big one)

```
Episode cost estimate (10 turns, 4K tokens avg per turn, 2 agents):
- Opus 4.7  ($15/$75 per M tokens)  : ~$0.40-0.60 per episode
- GPT-5.2   ($10/$30 per M tokens)  : ~$0.20-0.30 per episode
- Gemini 3  ($7/$21 per M tokens)   : ~$0.15-0.20 per episode

v0.1 evaluation set:
- 200 scenarios × 5 cultures × 3 seeds = 3,000 episodes per model
- 6 models tested = 18,000 episodes
- ~$0.30 avg/episode = ~$5,400 in API costs
```

### Mitigation plan

| Issue | Mitigation | Implementation |
|---|---|---|
| $5K API spend | Apply for research credits FIRST. OpenAI Researcher Access, Anthropic Research API credits, Google AI Studio free tier. Realistic to land $1-3K free. | Week 1, before any API spend. |
| Re-run cost | Cache LLM calls keyed on (model, prompt_hash, temp, seed). | `concord/runners/cache.py` from day 1. |
| Rate limits at burst | Async runner with semaphore-bounded concurrency. ~30 concurrent per provider safe. | `concord/runners/run_batch.py` |
| Cost overrun | Daily budget caps in runner. Hard stop at $X. | `concord/runners/budget.py` |
| Failed episodes | Dead-letter queue + retry with exponential backoff. Save partial transcripts. | `concord/runners/run_batch.py` |

### 7.2 Synthetic generation throughput

AWM generates ~50 scenarios per hour per model on the synthesis side. For v0.1 target of 5K+ scenarios, that's a one-day batch. Not a bottleneck.

### 7.3 Episode concurrency

- Async Python runner with `asyncio.gather` + semaphores
- ~30 concurrent episodes per provider (rate-limit safe)
- ~2-min average episode length
- 18K episodes ≈ 20 hours wall time
- Plan as overnight runs, not interactive

### 7.4 Storage

```
JSONL transcripts:    ~50KB each × 18K episodes = ~900MB
SQLite scenario state: ~5MB total for 5K scenarios
Parquet aggregates:    ~10MB
Activation logs:       ~100MB per open-weight experiment

Total v0.1: ~1-2GB
```

Push transcripts to GitHub LFS or HF dataset on completion. Don't commit to git proper.

### 8.4 Conflict flags

- **Lanes A, B, C all touch `concord/synth/`** — but as separate files (`enrichment.py`, `cultural_adapter.py`, `repeated_game.py`). No conflict if module split is respected.

If treating Concord as a side project at 10-15 hr/week, parallelism is mostly theoretical — work one lane at a time. The lane structure is a cheat sheet for which agents can run in parallel via Claude Code worktrees during sprint sessions.

## 10. Reproducibility Protocol

Before any public results:

1. **Pin model snapshots:** Always specify exact API model IDs with snapshot dates.
   - `claude-opus-4-7` (specific snapshot)
   - `gpt-5.2-2026-04-15` (or whatever the snapshot date is)
   - `gemini-3-pro-2026-04` (specific version)

2. **Log per-episode:**
   - prompt_hash (SHA256 of system prompt + user prompt)
   - temperature, top_p, max_tokens
   - random seed
   - model_id (exact snapshot)
   - timestamp
   - concord version tag

3. **Version-tag every published result:**
   - Tag in git: `v0.1.0`, `v0.1.1`
   - Each paper / blog post / leaderboard entry references the tag
   - Re-running the same tag with the same model snapshots reproduces the same results (modulo provider non-determinism)

4. **Document non-determinism:**
   - Some providers are non-deterministic even with seed (provider-side batching). Note this in the paper.
   - Run 3 seeds per scenario; report mean + 95% bootstrap CI.
